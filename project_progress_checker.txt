Virtual Gesture Mouse - Development Progress Checklist

## Global Pipeline Constraint

**Goal:** Enforce a chronological, implementation-safe ML and runtime pipeline

* [ ] Enforce strict pipeline order: Recording -> Data Validation -> Feature Validation -> Split Enforcement -> Training -> Evaluation -> Runtime
* [ ] No module may consume artifacts from a future phase
* [ ] All data must pass validation before training or inference
* [ ] Training, validation, test, and runtime data flows remain strictly separated
* [ ] No data leakage is permitted across users, sessions, or dataset splits
* [ ] Runtime control logic may consume only exported evaluation-approved model artifacts

**Validation rule:**
No model training, auth sequencing, or OS control work may be marked complete unless all prerequisite checklist items in earlier phases are complete.

## Phase 6 - Mouse & Presentation Control

**Goal:** Safe OS interaction with app-aware modes

Implementation status:
- General-mode dry run contract now exists: `app/control/actions.py` defines non-OS `ActionIntent`, `app/modes/general.py` returns explicit dry-run/no-op intents, and `app/lifecycle/runtime_loop.py` surfaces them in overlay text
- `PINCH_INDEX` primary interaction dry run now exists: stateful candidate -> drag -> click-pending -> single/double-click resolution using abstract cursor-space movement, decoupled from the final cursor pose
- `FIST` clutch dry run now exists above primary interaction: it freezes General Mode ownership, cancels lower-priority primary state, and suppresses same-frame re-entry on release
- `PINCH_MIDDLE` secondary interaction dry run now exists as a sibling release-only controller: valid release -> one right click, excessive drift/hand loss/clutch -> cancel
- `PINCH_IM` scroll-mode dry run now exists as a mode controller: edge-triggered toggle, dead-zone entry, deterministic axis lock/reset, and suppression of primary/secondary while active
- General Mode now has a provisional-but-isolated cursor policy layer plus non-OS cursor preview controller beneath the higher-priority subsystems
- Safe OS execution seam now exists behind the dry-run contract; live cursor + primary click/drag/double-click + right click + scroll now translate from dry-run outputs through one gated adapter
- Presentation mode now runs on the same dry-run -> safety -> execution architecture: `WindowWatch` provides conservative foreground-app context, `app/modes/presentation.py` resolves presentation intents, and the executor translates approved presentation actions into key presses when enabled
- Cursor pose choice remains intentionally provisional and isolated; live OS execution now uses an explicit `dry_run`/`live` profile, per-subsystem toggles, and runtime safety suppression, and stays disabled by default
